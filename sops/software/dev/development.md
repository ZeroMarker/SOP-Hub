## 1. 目的

规范软件开发全流程，确保代码质量、交付效率与团队协作效率，建立可追溯的开发规范，降低项目风险。

## 2. 适用范围

适用于公司所有软件产品及项目的开发、测试、部署与维护活动，涵盖需求分析、设计、编码、测试、发布全生命周期。

## 3. 术语与定义

| 术语 | 定义 |
|------|------|
| 需求评审 | 对需求文档进行集体审议，确认需求完整性、合理性与可实现性的过程 |
| 代码评审 | 由其他开发人员对提交的代码进行审查，确保代码质量与规范性的过程 |
| Sprint | 敏捷开发中的固定迭代周期，通常为 1-4 周 |
| CI/CD | 持续集成/持续部署，自动化构建、测试与发布流程 |
| 技术债 | 为了短期目标而采用的非最优解决方案，长期需要偿还的开发成本 |
| Code Review | 代码评审的英文表述 |
| Pull Request | 代码合并请求，开发者请求将代码分支合并到主分支的流程 |

## 4. 职责分工

### 4.1 产品经理
- 负责需求收集、分析与文档编写
- 组织需求评审会议
- 确认需求优先级与版本规划
- 验收功能实现是否符合需求

### 4.2 技术负责人/架构师
- 技术方案设计与评审
- 技术选型与架构决策
- 代码评审与质量把关
- 技术债务管理与优化

### 4.3 开发工程师
- 根据需求进行详细设计与编码
- 编写单元测试
- 参与代码评审
- 修复缺陷与优化性能

### 4.4 测试工程师
- 编写测试用例与测试计划
- 执行功能测试、集成测试、性能测试
- 缺陷跟踪与验证
- 测试报告编写

### 4.5 项目经理
- 项目计划制定与进度跟踪
- 资源协调与风险管理
- 跨团队沟通协调
- 项目文档管理

### 4.6 运维工程师
- 环境搭建与维护
- CI/CD 流水线配置
- 监控告警与故障响应
- 发布部署与回滚

## 5. 操作流程

### 5.1 需求阶段

```
需求收集 → 需求分析 → 需求评审 → 需求确认 → 需求排期
```

**详细步骤：**

1. **需求收集**：
   - 产品需求文档（PRD）编写
   - 用户故事（User Story）定义
   - 验收标准（Acceptance Criteria）明确
   - 优先级评定（P0/P1/P2/P3）

2. **需求评审**：
   - 召开需求评审会议
   - 参与方：产品、开发、测试、设计
   - 确认需求完整性、一致性、可测试性
   - 记录评审意见并修改完善

3. **需求确认**：
   - 需求文档定稿签字
   - 建立需求追踪矩阵
   - 拆解为可执行的开发任务

### 5.2 设计阶段

```
技术方案设计 → 设计评审 → 详细设计 → UI/UX 设计 → 设计定稿
```

**详细步骤：**

1. **技术方案设计**：
   - 系统架构设计
   - 接口设计（API 文档）
   - 数据库设计（ER 图、表结构）
   - 技术选型与依赖确认

2. **设计评审**：
   - 召开设计评审会议
   - 参与方：技术负责人、架构师、核心开发
   - 评审架构合理性、扩展性、安全性
   - 确认技术方案可行

3. **详细设计**：
   - 模块划分与职责定义
   - 关键流程时序图
   - 数据流设计
   - 异常处理方案

### 5.3 开发阶段

```
任务分配 → 环境准备 → 编码开发 → 单元测试 → 代码提交
→ 代码评审 → 合并分支
```

**详细步骤：**

1. **开发准备**：
   - 从主分支拉取最新代码
   - 创建功能分支（命名规范：`feature/{功能名}-{Issue号}`）
   - 配置本地开发环境
   - 确认依赖与配置

2. **编码规范**：
   - 遵循团队编码规范（见第 7 章）
   - 提交信息规范（Conventional Commits）
   - 每个功能模块编写必要的注释
   - 避免提交调试代码与敏感信息

3. **单元测试**：
   - 编写单元测试用例
   - 测试覆盖率要求：核心模块 ≥ 80%，一般模块 ≥ 60%
   - 测试通过后方可提交代码

4. **代码提交**：
   - 小批量、高频率提交
   - 提交信息格式：`<type>(<scope>): <description>`
   - type 类型：feat / fix / docs / style / refactor / test / chore

5. **代码评审（Code Review）**：
   - 提交 Pull Request（PR）
   - 指定至少 1 名评审人员
   - 评审要点：代码质量、逻辑正确性、安全性、性能
   - 评审意见必须全部处理后方可合并

### 5.4 测试阶段

```
测试计划 → 测试用例编写 → 测试用例评审 → 冒烟测试
→ 功能测试 → 缺陷修复 → 回归测试 → 测试报告
```

**详细步骤：**

1. **测试计划**：
   - 明确测试范围与测试策略
   - 确定测试环境与测试数据
   - 制定测试时间计划

2. **测试用例**：
   - 基于需求文档编写测试用例
   - 覆盖正常流程、异常流程、边界条件
   - 测试用例评审通过后执行

3. **测试执行**：
   - 冒烟测试：验证核心功能可用
   - 功能测试：全面验证各功能模块
   - 集成测试：验证模块间交互
   - 性能测试：压力测试、并发测试（如需）
   - 安全测试：漏洞扫描、渗透测试（如需）

4. **缺陷管理**：
   - 缺陷分级：致命 / 严重 / 一般 / 轻微
   - 缺陷必含：复现步骤、预期结果、实际结果、环境信息
   - 致命/严重缺陷必须修复后方可发布

### 5.5 发布阶段

```
发布申请 → 发布评审 → 预发布验证 → 正式发布 → 线上验证
→ 监控确认 → 发布完成
```

**详细步骤：**

1. **发布准备**：
   - 确认测试报告与验收通过
   - 准备发布清单（Release Checklist）
   - 变更日志（CHANGELOG）编写
   - 通知相关方发布时间

2. **发布评审**：
   - 确认发布内容与版本号
   - 确认回滚方案
   - 确认发布窗口期
   - 获得发布批准

3. **部署发布**：
   - 按 CI/CD 流水线执行部署
   - 数据库变更脚本执行（如有）
   - 配置变更同步
   - 灰度发布/蓝绿部署（视情况）

4. **发布验证**：
   - 核心功能冒烟验证
   - 监控指标确认（错误率、响应时间、资源使用）
   - 日志检查
   - 用户反馈渠道监控

### 5.6 运维阶段

```
监控告警 → 故障响应 → 问题定位 → 修复/回滚
→ 复盘总结 → 知识沉淀
```

**详细步骤：**

1. **监控告警**：
   - 应用监控：错误率、响应时间、吞吐量
   - 基础设施监控：CPU、内存、磁盘、网络
   - 业务监控：关键业务指标
   - 日志监控：异常日志告警

2. **故障响应**：
   - 告警触发后 5 分钟内响应
   - 初步评估故障级别与影响范围
   - 通知相关人员
   - 启动应急处理流程

3. **故障处理**：
   - 优先恢复服务（重启/回滚/降级）
   - 定位根本原因
   - 制定修复方案
   - 实施修复并验证

## 6. 特殊情况处理

### 6.1 紧急修复（Hotfix）
- 命名规范：`hotfix/{问题描述}-{Issue号}`
- 从生产分支拉取代码
- 修复后经技术负责人评审
- 简化测试流程，重点验证修复内容
- 合并回主分支与开发分支

### 6.2 需求变更
- 评估变更影响范围
- 变更审批流程
- 更新需求文档与任务列表
- 通知相关开发与测试人员
- 记录变更日志

### 6.3 技术债务
- 建立技术债务登记表
- 每个 Sprint 预留 10-20% 时间偿还技术债
- 技术债优先级评估
- 定期回顾与清理

### 6.4 线上故障
- 启动应急预案
- P0 故障：15 分钟内响应，1 小时内恢复
- P1 故障：30 分钟内响应，4 小时内恢复
- 故障复盘：24 小时内完成

## 7. 编码规范

### 7.1 通用规范
- 使用一致的代码缩进（推荐 2 或 4 空格）
- 变量、函数、类命名清晰有意义
- 避免魔法数字，使用常量
- 单个函数不超过 50 行
- 单个文件不超过 500 行

### 7.2 Git 提交规范
```
<type>(<scope>): <subject>

<body>

<footer>
```

| type | 说明 |
|------|------|
| feat | 新功能 |
| fix | 修复缺陷 |
| docs | 文档变更 |
| style | 代码格式（不影响逻辑） |
| refactor | 重构（不新增功能、不修复缺陷） |
| test | 测试相关 |
| chore | 构建/工具变更 |

### 7.3 分支管理
- `main` / `master`：生产分支，保持稳定
- `develop`：开发分支，集成最新功能
- `feature/*`：功能分支
- `hotfix/*`：紧急修复分支
- `release/*`：发布准备分支

## 8. 服务质量要求

| 项目 | 标准 |
|------|------|
| 代码评审覆盖率 | 100%（所有 PR 必须评审） |
| 单元测试覆盖率 | 核心模块 ≥ 80%，一般模块 ≥ 60% |
| 缺陷修复及时率 | 致命/严重 24h 内修复 |
| 发布成功率 | ≥ 99% |
| 生产故障响应时间 | P0 ≤ 5 分钟，P1 ≤ 30 分钟 |
| 需求交付周期 | ≤ 2 个 Sprint |
| 代码评审响应时间 | ≤ 4 小时 |
| 文档更新及时率 | 100%（代码变更同步更新文档） |

## 9. 项目管理

### 9.1 迭代管理
- 采用敏捷开发模式（Scrum / Kanban）
- Sprint 周期：2 周
- 每日站会：15 分钟内
- Sprint 评审与回顾会议

### 9.2 任务管理
- 使用项目管理工具（Jira / Tapd / 禅道等）
- 任务状态：待开发 → 开发中 → 待测试 → 测试中 → 待发布 → 已发布
- 任务估时与实际工时记录

### 9.3 文档管理
- 需求文档：产品需求文档（PRD）
- 技术文档：技术方案、接口文档、数据库设计
- 测试文档：测试计划、测试用例、测试报告
- 运维文档：部署手册、运维手册、故障处理手册

## 10. 信息安全管理

- 代码仓库权限分级管理
- 敏感配置使用环境变量，禁止硬编码
- 数据库密码、API Key 等不得提交至代码库
- 生产环境访问需审批
- 代码扫描与安全审计
- 开发环境与生产环境隔离
